Ethereum State Growth: The Hidden MPT Tax

TL;DR: Ethereum's state is 290 GB, but only 133 GB (46%) is actual data. The other 157 GB is Merkle Patricia Trie overhead. The MPT overhead ratio has grown from 2.08x to 2.09x over the past 6 months and is projected to worsen by 0.02x annually. Storage slots are surprisingly 82% more efficient than account state, suggesting contract-heavy usage patterns are actually slowing state bloat.

The 2x State Tax Nobody Talks About

When people discuss Ethereum state growth, they usually quote the headline number: 290 GB. But here's the thing - that's not the size of Ethereum's "data." It's the size of Ethereum's data plus all the cryptographic scaffolding required to make it verifiable.

The breakdown is stark:

Component Size (GB) % of Total
Account Data 17.6 6.1%
Account Trie Overhead 60.5 20.8%
Storage Data 115.2 39.6%
Storage Trie Overhead 217.5 74.8%
Contract Code included above -
Total State 290.8 100%

The "MPT overhead ratio" - trie bytes divided by actual data bytes - is currently 2.09x. Every gigabyte of real state data carries 2.09 gigabytes of Merkle proof overhead.

Stacked area chart showing how Ethereum's 290 GB state is composed. The red (storage trie) and orange (account trie) areas dominate, representing 95.6% of total state.

Getting Worse, Slowly

I tracked the MPT overhead ratio over 6 months (August 2025 to February 2026) to see if this is a static cost or a growing problem. It's growing.

Period MPT Overhead Ratio Trie % of State
August 2025 2.083x 95.61%
February 2026 2.093x 95.61%
Change (6 months) +0.010x ~0%

The overhead ratio increased by 0.01x over 179 days. Annualized, that's 0.02x per year. At this rate, by 2030 the MPT overhead ratio will be ~2.17x.

The MPT overhead ratio has climbed steadily from 2.08x to 2.09x over 6 months. The red dashed line shows the 2.0x threshold.

The Storage Efficiency Surprise

Here's something counterintuitive: storage slots are more efficient than accounts.

Data Type MPT Overhead Ratio Efficiency vs Average
Accounts 3.43x 64% worse
Storage Slots 1.89x 10% better
Combined 2.09x baseline

Storage slots have 1.89x overhead while accounts have 3.43x overhead - an 82% difference. This means contracts that use storage heavily (like ERC20 tokens, DeFi protocols) are actually more state-efficient per byte than simple EOAs.

Why? The account trie is sparser. Most Ethereum addresses have never been used, but they still occupy trie nodes. The storage trie is denser - contracts that use storage tend to fill contiguous slots, creating more efficient trie structures.

Growth Projections

Based on the last 6 months of data:

  • Current state growth: ~60 GB/year
  • Data portion: ~20 GB/year
  • Trie overhead portion: ~40 GB/year

Without MPT overhead, Ethereum's state would be 133 GB today instead of 291 GB. In 5 years at current growth rates:

  • Total projected state: ~590 GB
  • Actual data: ~233 GB
  • Trie overhead tax: ~357 GB (2.5x the data)

Why This Matters

The MPT overhead isn't just a disk space issue. It affects:

  1. Sync times: Full nodes must download and verify 291 GB, not 133 GB
  2. State access latency: More trie nodes = more disk seeks
  3. Verkle transition urgency: Verkle trees promise 5-10x reduction in proof sizes
  4. State expiry feasibility: The bigger state gets, the harder partial statelessness becomes

Trie nodes have remained steady at 95.6% of total state, fluctuating within a 0.04% band. The overhead is consistent.

Data & Methodology

Source: mainnet.fct_execution_state_size_daily (xatu-cbt cluster)

Date range: August 8, 2025 to February 3, 2026 (180 days)

Key Metrics Calculated

# MPT Overhead Ratio = trie bytes / data bytes
total_mpt_overhead_ratio = (account_trienode_bytes + storage_trienode_bytes) / (account_bytes + storage_bytes)

# Trie percentage of total state
trie_pct_of_total = (total_trie_bytes / total_bytes) * 100

Query

SELECT
  day_start_date,
  total_bytes,
  account_bytes,
  account_trienode_bytes,
  storage_bytes,
  storage_trienode_bytes
FROM mainnet.fct_execution_state_size_daily FINAL
WHERE day_start_date >= today() - 180
ORDER BY day_start_date ASC

Conclusions

What We Learned

  • Ethereum state is 95.6% trie overhead, 4.4% actual data
  • The MPT overhead ratio is slowly worsening by ~0.02x/year
  • Storage slots are 82% more efficient than accounts - contracts aren't the bloat problem
  • Without MPT overhead, state would be 54% smaller (133 GB vs 291 GB)

Limitations

  • 6-month window may not capture long-term trends
  • Contract code size is underreported in current schema
  • No granular breakdown of which contract types drive storage growth
  • Pre-Dencun comparison data unavailable due to query timeouts

Open Questions

  • Will the MPT overhead ratio accelerate as state gets larger?
  • How does Verkle tree efficiency compare for the same data?
  • Which specific contracts are driving storage growth?
